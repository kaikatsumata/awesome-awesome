# Awesome List 作成のベストプラクティスと調査手法

本ドキュメントは、AI研究分野の「リストのリスト(awesome-of-awesome)」を構築した本リポジトリの経験に基づき、
**(1) 一般的なベストプラクティス**、**(2) 本セッションで得た学び**、**(3) 採用した調査手法**を整理したものです。
新たに awesome list / サーベイ集を作る際の再利用可能な指針として残します。

---

## 1. 一般的なベストプラクティス

[awesome.re](https://awesome.re) / [sindresorhus/awesome](https://github.com/sindresorhus/awesome) のガイドラインに準拠した、
広く通用する作法。

### 命名・体裁
- リポジトリ名・README先頭は `Awesome <Topic>`。GitHub Topics に **`awesome`** を必ず付与する
  （[github.com/topics/awesome](https://github.com/topics/awesome) からの発見性が上がる）。
- README 先頭に Awesome バッジ `[![Awesome](https://awesome.re/badge.svg)](https://awesome.re)` を置く。
- 冒頭に1〜2文の **簡潔な説明**（何を・誰のために集めたか）を置く。

### 構造
- **目次(TOC)** を用意し、各カテゴリへアンカーリンクする。
  - GitHubのアンカー生成規則（小文字化・記号除去・空白→ハイフン、絵文字は除去され前後のハイフンが残る）を踏まえる。
- **意味のあるカテゴリ分け**を行い、各項目は1行で「何であるか」を説明する（リンクだけを並べない）。
- 重複を排除する。1項目は原則1カテゴリに正規化して置く。
- アルファベット順 or 重要度/活発度順など、**並び順の方針を一貫**させる。

### 品質・運用
- **リンク切れ(404)を残さない**。定期的に死活確認する。
- **リダイレクト（リネーム/移管）は正規URLに解決**してから掲載する。
- `awesome-lint` でフォーマット検証ができる。
- `contributing.md`（貢献ガイド）・`code-of-conduct.md` を用意すると公式 awesome への掲載要件を満たしやすい。
- ライセンスは **CC0（パブリックドメイン）** が awesome list の慣例。
- 継続更新できる仕組み（後述の自動化）を持つ。

## 2. 本セッションで得た学び（AI研究リスト特有）

実際に約900件の研究リストを収集・分類する中で有効だった、一般論より一歩踏み込んだ知見。

### 鮮度（最終更新日・更新頻度）を第一級の情報として扱う
- 研究リストは**陳腐化が速い**。star数だけでなく **最終更新日(`pushed_at`)** を必ず併記し、
  鮮度マーカー（🟢活発 / 🟡中程度 / 🔴停滞）で一目で分かるようにする。
- 並び順も「鮮度 → star数」にすると、読者がまず生きているリストに辿り着ける。

### ただし「古い＝低品質」ではない。例外を設ける
本セッションで明確になった重要な例外（鮮度ペナルティを免除すべきもの）:
- **📑 査読付きサーベイ論文の companion リポジトリ**: TPAMI/IJCV/CSUR/TKDE 等に採録された
  サーベイ論文に紐づくリポジトリは、更新が止まっていても **網羅性・権威性・キュレーション品質**が高く有用。
  停滞🔴ではなく専用マーカーで示す。
- **📚 歴史的・古典コレクション**: その分野の基礎を成す定番（例: 著名な `awesome-computer-vision`、
  被引用論文Top100、学習ロードマップ）は、更新停止が前提でも残す価値がある。

### 「リスト/サーベイ」と「モデル本体/ツール」を峻別する
- 公式モデルの推論リポジトリ（例: `flux`, `Wan2.1`, `HunyuanVideo`）や単なるライブラリは
  **作例・プロンプトのコレクション(リスト)ではない**。awesome-of-awesome の対象から除外する。
- 一方、`awesome-flux` のような**エコシステムのキュレーション**は対象。混同しない。
- 特定モデルの作例集（`awesome-nanobanana-pro` 等）はリストなので対象に含める。

### 種別(type)を明示して期待値を揃える
- `awesome-list`（キュレーション） / `survey`（査読サーベイ付随） / `paper-list`（論文リスト） /
  `model-collection`（特定モデルの作例集）を区別すると、読者が中身を予測できる。

### 重複排除は「正規名」で行う
- リダイレクトを解決した後の **`full_name`（正規名）で重複判定**する。
  別名・旧名で同一リポジトリを二重掲載しないため。

### ドキュメントを分離する
- **READMEは分類済みの一覧**に徹し、**全調査結果・統計・メタデータ**は別ドキュメント
  （本リポジトリでは [`research-notes.md`](research-notes.md)）に分ける。READMEの可読性を保てる。

### データ駆動 + 自動化で保守可能にする
- 出典(JSON) → メタデータ補強(GitHub API) → README/ノート生成、という**生成パイプライン**にすると、
  鮮度更新・項目追加・体裁変更がスクリプト再実行だけで反映できる。手作業のMarkdown編集は破綻しやすい。
- 鮮度は **GitHub Actions の定期実行(cron)** で自動更新し、変化があればコミットする。

## 3. 採用した調査手法（網羅性をどう担保したか）

「新しいリストが見つからなくなるまで」探索するための、再現可能な手順。

### (A) 分野ごとの並列探索（fan-out）
- CV / CG / NLP / RL / 生成AI / 音声 / ロボ / GNN / … と**分野を分担した複数エージェントを並列起動**し、
  各々が WebSearch/WebFetch + GitHub API で実在確認しながら収集。1巡で数百件を一気に集める。

### (B) 反復（波）で逓減を観測し、飽和で止める
- 第1波: 主要分野 → 第2波: ニッチ細分（Mamba/KAN・メタ学習・因果 等） → 第3〜4波: さらに細分・新興・特定モデル。
- 各波の**新規追加数の逓減**（例: +400 → +190 → +110 → +72 → +62 → +41）を見て、
  「主要分野の大きな抜けが無い」ことをメタ確認エージェントに判定させて終了する。

### (C) GitHub Topics / Search からの掃き出し
- `topic:awesome-list` / `topic:paper-list` / `topic:survey` / `topic:papers` を **高star条件**で走査し、
  未収録の有力リストを発見する。
- 注意: `topic:survey` は**サーベイ調査ソフト(form builder)**を多く拾うなど誤検出が出るため、
  名称・説明でフィルタする。

### (D) prolific な著者・org の深掘り
- 良質なサーベイ/awesomeを**量産する個人・組織**（例: 幾何処理の `Bigger-and-Stronger`、
  会議論文の `amusi`/`DmitryRyumin`、清華大系 `thunlp`/`THUDM`、`opendilab` 等）を特定し、
  その**リポジトリ一覧をAPIで列挙**して list 系を一括抽出する。1つ良いリストを見つけたら作者を辿る。

### (E) 査読サーベイ companion をカテゴリとして掃き出す
- 「`X: A Survey` のプロジェクトリポジトリ」というパターン自体を狙って探すと、
  古くても高品質なキュレーションが体系的に集まる。

### (F) 実在確認と正直な報告
- 収集物は必ず **GitHub API で実在・star・最終更新を検証**（捏造URLを排除、404を除外）。
- 「専用の決定版リストが存在しない」分野（例: 多腕バンディット、AR/VR×AI、記号回帰、音源分離専用）は
  **無理に水増しせず、存在しないことを明記**する。網羅性の主張の誠実さを保つ。

## まとめ（チェックリスト）

新しく awesome list を作る/維持するときの要点:

- [ ] `Awesome <Topic>` 命名・Awesomeバッジ・`awesome` トピック付与
- [ ] 目次・カテゴリ分け・1行説明・並び順方針の一貫性
- [ ] star数だけでなく **最終更新日と鮮度マーカー** を併記
- [ ] **査読サーベイ(📑)・歴史的定番(📚)** は鮮度ペナルティを免除
- [ ] **リスト/サーベイ と モデル本体/ツール を峻別**して対象を絞る
- [ ] **正規名で重複排除**、リンク切れ(404)を残さない
- [ ] **READMEと調査ノートを分離**
- [ ] **データ駆動の生成パイプライン + 定期自動更新(Actions)**
- [ ] 分野fan-out → 反復で飽和確認 → Topics掃き出し → prolific org深掘り、で網羅性を担保
- [ ] 実在検証と「存在しない分野の明記」で誠実さを保つ
